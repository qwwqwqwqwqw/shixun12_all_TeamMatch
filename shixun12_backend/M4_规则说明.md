# M4 模块规则说明（答辩材料）

> 面向答辩评委，说明 M4 模块的核心业务规则、状态机设计与退出机制。

---

## 1. 模块职责边界

M4 负责 TeamMatch 平台的**项目管理与组队协作**核心业务，共五大功能域：

| 功能域 | 内容 |
|--------|------|
| 项目管理 | 创建、列表、详情、更新、状态流转 |
| 组队请求 | 邀请/申请、接受/拒绝/取消、列表查询 |
| 成员管理 | 成员列表、角色权限、满员/状态拦截 |
| 退出机制 | 主动退出（self_exit）、投票退出（exit_vote） |
| 信誉规则 | delta 映射服务、单项目 ±10 封顶服务（供 M5 调用） |

---

## 2. 项目状态机

### 2.1 状态流转图

```
recruiting（招募中）
    │  条件：队长调用 startProject；系统批量将所有 pending 请求设为 expired
    ▼
in_progress（进行中）
    │  条件：队长调用 endProject；系统写入 ended_at 和 eval_deadline（默认+7天）
    ▼
ended（已结束，互评窗口开放）
    │  条件：eval_deadline 到期（懒检查，下次查询时触发）
    ▼
eval_closed（互评已关闭）
```

### 2.2 关键规则

| 规则 | 说明 |
|------|------|
| 只有队长可以 start/end 项目 | 接口校验 `team_member.role = 'leader'` |
| 开始项目时失效所有 pending 请求 | 防止进行中项目仍有悬挂邀请/申请 |
| eval_deadline 采用懒检查 | 不用定时任务，查询时判断是否需要切换到 eval_closed |
| 状态不可逆 | 任何状态均不可回退到上一状态 |

---

## 3. 组队请求状态机

### 3.1 状态流转图

```
pending（待处理）
    ├── 接受方 accept  → accepted（自动写入 team_member，role=member）
    ├── 接受方 reject  → rejected
    ├── 发送方 cancel  → cancelled
    └── 项目 start     → expired（批量失效）
```

### 3.2 关键规则

| 规则 | 实现 |
|------|------|
| 同一用户对同一项目只能有一条 pending 请求 | 接口校验 `DUPLICATE_PENDING_REQUEST` |
| 项目满员时不可接受新请求 | 接口校验 `PROJECT_FULL`（`team_member.active_count >= max_members`） |
| 仅 recruiting 状态的项目可收发请求 | 接口校验 `PROJECT_NOT_RECRUITING` |
| 队长不可接受自己发出的邀请 | 业务校验，队长已是 active 成员 |
| 接受邀请/申请后，同目标的其他 pending 请求自动失效 | 防止重复加入 |

## 4. 退出机制

### 4.1 主动退出（self_exit）

**触发条件：** 成员本人在项目 `in_progress` 阶段调用退出接口。

**执行流程：**
1. 校验成员状态为 `active`，项目状态为 `in_progress`
2. 使用 CAS 更新 `team_member.status = 'exited'`（`WHERE status = 'active'`）
3. CAS 成功后调用 M5 写入 `credit_change`（`change_type=self_exit, change_value=-10`）
4. 更新 `user.credit_score -= 10`

**并发保护：** CAS 原子更新，`memberRows != 1` 时返回 `TEAM_VOTE_CONFLICT` 并不写流水。

### 4.2 投票退出（exit_vote）

**触发条件：** 队长认为某成员无法正常参与项目，发起投票请求其他成员表决。

**流程与规则：**

```
① 队长发起投票（initiateVote）
   - 校验：队长身份 + 目标成员 active + 无重复投票（uk_exit_vote_active_target）
   - 写入 exit_vote（status=voting, deadline_at = now + 24h）
   - penalty_level 由队长指定：negotiated（协商）/ malicious（恶意）
   ↓
② 成员提交投票（submitVote，24h 内）
   - 防重复：uk_exit_vote_record_voter 唯一键拦截
   - choice：agree / disagree
   ↓
③ 队长关闭投票（closeVote）
   - 事务 + 幂等：exit_vote.status='closed' 则直接返回
   - 统计：agree_count / total_voters
   - agree_count > total_voters / 2 → result=pass
   - pass 时：CAS 更新 team_member.status=exited + 写 credit_change(-10)
   - reject 时：仅更新 exit_vote.status，成员继续留队
```

**扣分规则：**

| 场景 | change_type | change_value |
|------|-------------|-------------|
| 主动退出 | `self_exit` | -10 |
| 被投票踢出（pass） | `exit_vote` | -10 |
| 投票未通过（reject） | — | 不扣分 |

### 4.3 并发保护设计

| 场景 | 保护机制 |
|------|---------|
| 两个请求同时关闭同一投票 | `@Transactional` + 幂等检查（status='closed'直接返回） |
| 两个请求同时更新同一 team_member | CAS（`UPDATE ... WHERE status='active'`），rows!=1 抛异常不写流水 |
| 重复发起对同一目标的投票 | 生成列 `active_vote_key` + `UNIQUE KEY` 数据库层拦截 |
| 重复提交同一投票 | `uk_exit_vote_record_voter` 唯一键拦截 |

## 5. 信誉分计算规则（供 M5 调用）

### 5.1 四维均分 → delta 映射（CreditRuleService.mapAverageScoreToDelta）

| 均分区间 | delta |
|---------|-------|
| [4.5, 5.0] | +5 |
| [4.0, 4.5) | +3 |
| [3.5, 4.0) | +1 |
| [3.0, 3.5) | 0 |
| [2.5, 3.0) | -1 |
| [2.0, 2.5) | -2 |
| [1.5, 2.0) | -3 |
| [1.0, 1.5) | -5 |

### 5.2 单项目 ±10 封顶（CreditRuleService.calculateCappedDelta）

同一用户在同一项目内，所有互评 delta 之和不得超过 ±10。

公式：`cappedDelta = max(-10, min(10, currentSum + newDelta)) - currentSum`
