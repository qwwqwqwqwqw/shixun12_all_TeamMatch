# M3 板块二：技术画像与排行榜设计说明

## 1. 概述

技术画像与排行榜是 TeamMatch 冷启动策略的核心组件，属于 **M3 认证与档案模块的板块二**。

**目标**：在平台上线初期，通过 GitHub / Gitee 公开数据分析生成用户技术画像，构建技术排行榜，解决"零用户时无内容可看"的冷启动问题。用户浏览排行榜后，通过绑定 GitHub 或 Gitee → 认领画像 → 完善档案的链路，转化为平台的正式用户。

```
排行榜引流 → 临时技术画像 → 学校邮箱认证 → 档案认领 → 进入正式榜单/搜索 → 发布或加入项目
```

**数据来源**：
- **GitHub**：通过 GitHub REST API 获取公开仓库、Stars、语言等数据；手动绑定方式无需授权
- **Gitee**：通过 Gitee OAuth 授权获取用户仓库访问权限，可获取真实 commits 和 PRs 数据；也支持手动绑定（通过公开 API 验证用户名）

## 2. 数据库设计

### 2.1 tech_profile 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 主键 |
| `github_username` | VARCHAR(64) | GitHub / Gitee 用户名 |
| `source` | VARCHAR(32) | 数据来源：`github` 或 `gitee` |
| `claimed_by_user_id` | BIGINT NULL | 认领该画像的用户 ID，NULL 表示未认领 |
| `total_stars` | INT | 所有原创仓库的 Star 总数 |
| `total_repos` | INT | 公开仓库数量（不含 fork） |
| `total_commits` | INT | 总提交数 |
| `total_prs` | INT | Pull Request 数 |
| `total_contributions` | INT | 贡献数（估算值） |
| `top_languages` | VARCHAR(500) | 主力编程语言 Top 5，JSON 数组格式 |
| `tech_score` | INT | 综合技术评分（排行榜排序依据） |
| `bio` | VARCHAR(500) | GitHub 个人简介 |
| `avatar_url` | VARCHAR(512) | GitHub 头像 URL |
| `last_synced_at` | DATETIME | 最后一次从 GitHub 同步的时间 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

**索引**：
- `uk_tech_profile_username_source` — (github_username, source) 联合唯一索引，同一平台用户名唯一
- `idx_tech_profile_claimed_user` — 按认领用户查询
- `idx_tech_profile_tech_score` — 按技术评分排序（排行榜核心索引）

### 2.2 与 user 表的关联

`user` 表新增字段 `tech_profile_id`（外键 → `tech_profile.id`），建立用户与技术画像的一对一关联。

认领时双向写入：
- `tech_profile.claimed_by_user_id = userId`
- `user.tech_profile_id = techProfileId`

## 3. 技术评分算法

### 3.1 评分公式

```
techScore = totalStars × 10 + totalCommits × 2 + totalPRs × 5 + totalRepos × 3 + totalContributions × 1
```

### 3.2 权重说明

| 指标 | 权重 | 理由 |
|------|------|------|
| Stars | ×10 | 最高权重。Star 是社区对项目质量最直接的认可 |
| PRs | ×5 | 反映协作能力与开源贡献活跃度 |
| Repos | ×3 | 项目多样性 |
| Commits | ×2 | 代码产出量 |
| Contributions | ×1 | 基础活跃度指标 |

权重设计参考了开源社区常用的"GitHub Score"非官方排名惯例。

### 3.3 数据来源与限制

由于本平台使用 **无认证的 GitHub REST API**（速率限制：60 次/小时），数据获取存在以下处理策略：

| 数据 | 获取方式 | 说明 |
|------|---------|------|
| Stars / Forks / Repos / 语言 | `GET /users/{username}/repos` | 直接获取，跳过 fork 仓库 |
| 用户基本信息 | `GET /users/{username}` | 获取 bio、avatar、public_repos |
| Commits | **估算**：`stars×20 + forks×10` | GitHub API 不提供跨仓库总 commit 数 |
| PRs | **估算**：`repos×2` | 无认证无法查询 PR 历史 |
| Contributions | **估算**：`stars + repos×10` | 精确值需要认证后的 GraphQL API |

> **注意**：估算值会低估实际贡献量。对评分的影响是整体性的（所有人被低估的程度相似），因此排名的相对公平性不受明显影响。

## 4. API 接口

### 4.1 技术画像接口

| 方法 | 路径 | 认证 | 说明 |

| `GET` | `/api/profile/tech-profile/{userId}` | 不需要 | 获取指定用户的技术画像 |

```json
    "totalPrs": 38,
    "totalContributions": 450,

    "claimed": true
  }

```

**错误码**：
- `M3026` — 技术画像不存在，请先绑定 GitHub 账号

### 4.2 排行榜接口

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| `GET` | `/api/leaderboard?page=1&size=20` | 不需要 | 分页获取排行榜 |
| `GET` | `/api/leaderboard/count` | 不需要 | 获取排行榜总条目数 |

**请求参数**：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `page` | int | 1 | 页码（从 1 开始） |
| `size` | int | 20 | 每页条数（上限 100） |

**响应结构 (LeaderboardEntryVO)**：

```json
  "data": [
    {
      "rank": 1,
      "userId": 2,
      "nickname": "张三",
      "avatarUrl": "https://...",

      "githubUsername": "zhangsan-github",
      "techScore": 5128,
      "totalStars": 245,
      "totalRepos": 12,
      "totalCommits": 1523,
      "totalPrs": 38,
      "totalContributions": 450,
      "topLanguages": "[\"Java\",\"Python\",\"TypeScript\"]",

      "rank": 2,
      "userId": null,
```

**榜单规则**：
- 按 `tech_score` 降序排列
- 已认领的画像附带用户信息（nickname、school、avatarUrl）
- 未认领的画像也展示在榜单中（motivates 认领），但 `userId`/`nickname` 为 null

## 5. GitHub 绑定与认领流程

### 5.1 接口

**绑定**：`POST /api/profile/github/bind`  
**Body**：`{ "githubUsername": "your-github-id" }`  
**认证**：需要（Bearer Token）

### 5.2 内部流程

```
POST /api/profile/github/bind { githubUsername: "zhangsan" }
    │
    ├─ 1. 更新 user 表：github_username + github_claimed = true
    │
    └─ 2. 调用 TechProfileService.claimProfile()
         │
         ├─ 搜索 tech_profile 表
         │    ├─ 找到 & 未认领 → 直接认领 ✓
         │    ├─ 找到 & 已被他人认领 → 抛出异常 M3027 ✗
         │    └─ 未找到 → 走 GitHub API 生成
         │         │
         │         ├─ GET https://api.github.com/users/{username}
         │         ├─ GET https://api.github.com/users/{username}/repos?per_page=100
         │         ├─ 解析聚合：stars / forks / languages
         │         ├─ 计算 techScore
         │         ├─ INSERT 到 tech_profile
         │         └─ 认领 ✓
         │
         └─ 3. 更新 user.tech_profile_id 关联画像
```

**错误场景**：

| 场景 | 处理 |
|------|------|
| GitHub API 调用超时/失败 | 创建空画像占位 + syncStatus=failed，不阻断绑定流程 |
| 该 GitHub 账号已被其他用户认领 | 返回 `M3027` 错误 |
| GitHub 用户名不存在 | API 返回 404，创建空画像 |

### 5.3 何时调用

用户在小程序端访问"认领技术画像"页面（`pages/profile-claim/profile-claim`）时触发。前端应先确保用户：① 已登录、② 已验证学校邮箱、③ 已完善个人档案，然后调用 `POST /api/profile/github/bind` 完成认领。

## 6. 排行榜生成逻辑

### 6.1 数据来源

排行榜**完全从 `tech_profile` 表查询**，不依赖 M5 互评数据。原因是：
- 冷启动阶段互评数据可能为空
- 技术画像反映的是客观 GitHub 数据，更适合作为"引流"排序依据
- 互评信誉分是另一个维度，不应与技术能力混淆

### 6.2 排名规则

```sql
SELECT * FROM tech_profile ORDER BY tech_score DESC LIMIT {size} OFFSET {offset}
```

- 降序排列，techScore 越高排名越靠前
- 分页由 Spring 的 MyBatis-Plus Page 插件处理
- `rank` 字段在服务层计算（`(page-1) × size + 当前行号`）

### 6.3 与信誉分的关系

技术排行榜和技术画像 **不与信誉分 (credit_score) 混合计算**。两者的定位不同：

| 维度 | 技术画像 | 信誉分 |
|------|---------|--------|
| 数据来源 | GitHub API | 互评、退出投票、举报处罚 |
| 反映 | 技术能力 | 协作信用 |
| 模块 | M3 板块二 | M5 互评 |
| 用途 | 排行榜引流、技术匹配 | 组队准入、处罚依据 |

## 7. 接口调用示例

### 7.1 绑定 GitHub（含自动认领）

```bash
curl -X POST http://localhost:8080/api/profile/github/bind \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_TOKEN>" \
  -d '{"githubUsername": "your-github-id"}'
```

### 7.2 获取排行榜 Top 10

```bash
curl http://localhost:8080/api/leaderboard?page=1&size=10
```

### 7.3 查看当前用户技术画像

```bash
curl http://localhost:8080/api/profile/tech-profile \
  -H "Authorization: Bearer <YOUR_TOKEN>"
```
