# M3 认证模块使用指南

## 概述

M3 模块提供了完整的认证解决方案，包括：
1. **AuthUtil** - 认证工具类，简化 Controller 代码
2. **AuthExceptionHandler** - 全局异常处理器，统一处理认证异常

本文档详细说明如何使用这两个组件。

---

## 一、AuthUtil 工具类

### 1.1 三种方法对比

| 方法 | 返回值 | Token 无效时 | 适用场景 |
|------|--------|-------------|---------|
| `requireUserId(token)` | `Long` | 抛出 `AuthenticationException(M3000)` | **必须登录**的接口 |
| `getCurrentUserId(token)` | `Long` 或 `null` | 返回 `null` | **可选登录**的接口 |
| `requireAdmin(token)` | `void` | 抛出 `AuthenticationException(M3000)` 或 `AuthorizationException(M3009)` | **管理员权限**校验 |

### 1.2 使用示例

#### 方式 1：使用 `requireUserId()` - 必须登录（推荐）

```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {
    
    @Autowired
    private AuthUtil authUtil;
    
    @Autowired
    private OrderService orderService;
    
    /**
     * 获取我的订单（必须登录）
     */
    @GetMapping("/my")
    public Result<List<Order>> getMyOrders(@RequestHeader("Authorization") String token) {
        // 必须登录，token 无效时抛出 AuthenticationException(M3000)
        // 异常会被全局异常处理器捕获，自动返回 M3000
        Long userId = authUtil.requireUserId(token);
        
        // 能走到这里，说明一定已登录
        List<Order> orders = orderService.getUserOrders(userId);
        return Result.success(orders);
    }
}
```

**优点**：
- ✅ 代码简洁，无需手动判断 null
- ✅ 异常由全局处理器统一处理
- ✅ 语义明确："这个接口必须登录"

---

#### 方式 2：使用 `getCurrentUserId()` - 可选登录

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {
    
    @Autowired
    private AuthUtil authUtil;
    
    @Autowired
    private ProductService productService;
    
    /**
     * 浏览商品列表（登录用户看个性化推荐，未登录看热门商品）
     */
    @GetMapping
    public Result<List<Product>> getProducts(
            @RequestHeader(value = "Authorization", required = false) String token) {
        
        // 尝试获取用户 ID，不强制要求登录
        Long userId = authUtil.getCurrentUserId(token);
        
        List<Product> products;
        if (userId != null) {
            // 已登录：根据喜好推荐
            products = productService.getPersonalizedProducts(userId);
        } else {
            // 未登录：返回热门商品
            products = productService.getHotProducts();
        }
        return Result.success(products);
    }
}
```

**优点**：
- ✅ 灵活，支持"登录增强体验"的场景
- ✅ 不会因为未登录而拒绝服务
- ✅ 适合公开内容 + 个性化功能的混合场景

---

#### 方式 3：使用 `requireAdmin()` - 管理员权限校验

```java
@RestController
@RequestMapping("/admin/users")
public class UserAdminController {
    
    @Autowired
    private AuthUtil authUtil;
    
    @Autowired
    private UserService userService;
    
    /**
     * 删除用户（仅管理员）
     */
    @DeleteMapping("/{id}")
    public Result<Void> deleteUser(@PathVariable Long id,
                                   @RequestHeader("Authorization") String token) {
        
        // 先验证登录，再验证管理员权限
        // - Token 无效 → AuthenticationException(M3000)
        // - 非管理员 → AuthorizationException(M3009)
        authUtil.requireAdmin(token);
        
        userService.deleteUser(id);
        return Result.success(null);
    }
}
```

**优点**：
- ✅ 清晰区分两种错误：
  - M3000：你都没登录，怎么当管理员？
  - M3009：你登录了，但不是管理员
- ✅ 前端可以根据不同错误码给出不同提示
- ✅ 安全审计时可以记录谁尝试访问管理员接口

---

### 1.3 方法选择建议

| 场景 | 推荐方法 | 原因 |
|------|---------|------|
| 个人中心、创建资源 | `requireUserId()` | 必须知道用户身份 |
| 浏览公开内容 | `getCurrentUserId()` | 未登录也能用，登录后体验更好 |
| 后台管理、敏感操作 | `requireAdmin()` | 需要严格的权限校验 |

## 二、AuthExceptionHandler 全局异常处理器

### 2.1 功能说明

`AuthExceptionHandler` 是 M3 提供的全局异常处理器，自动捕获认证相关异常并返回统一的错误响应。

**处理的异常类型**：
- `MalformedJwtException` - Token 格式错误
- `ExpiredJwtException` - Token 已过期
- `SignatureException` - Token 签名验证失败
- `AuthenticationException` - 认证失败（M3000）
- `AuthorizationException` - 权限不足（M3009）

**返回结果**：
- JWT 异常 → `M3000 UNAUTHORIZED`
- AuthenticationException → `M3000 UNAUTHORIZED`
- AuthorizationException → `M3009 ADMIN_REQUIRED`

**重要**：不再全局捕获 `RuntimeException`，避免误捕其他模块的业务异常。

### 2.2 工作原理

```java
@Slf4j
@RestControllerAdvice
public class AuthExceptionHandler {
    
    // 1. 捕获 JWT 相关异常
    @ExceptionHandler({MalformedJwtException.class, ExpiredJwtException.class, SignatureException.class})
    public Result<Void> handleJwtException(Exception e) {
        log.warn("JWT Token 解析失败: {}", e.getMessage());
        return Result.fail(ReasonCode.UNAUTHORIZED);
    }
    
    // 2. 捕获认证失败异常（M3000）
    @ExceptionHandler(AuthenticationException.class)
    public Result<Void> handleAuthenticationException(AuthenticationException e) {
        log.warn("认证失败: {}", e.getReasonCode());
        return Result.fail(e.getReasonCode());
    }
    
    // 3. 捕获权限不足异常（M3009）
    @ExceptionHandler(AuthorizationException.class)
    public Result<Void> handleAuthorizationException(AuthorizationException e) {
        log.warn("权限不足: {}", e.getReasonCode());
        return Result.fail(e.getReasonCode());
    }
}
```

**设计要点**：
- ✅ 只处理认证相关的特定异常，不捕获通用的 RuntimeException
- ✅ 添加 warn 日志，方便排查和安全审计
- ✅ 区分认证失败（M3000）和权限不足（M3009）

### 2.3 使用场景

#### 场晩 1：直接使用 AuthUtil（推荐）

```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {
    
    @Autowired
    private AuthUtil authUtil;
    
    @Autowired
    private OrderService orderService;
    
    @GetMapping("/my")
    public Result<List<Order>> getMyOrders(@RequestHeader("Authorization") String token) {
        // 必须登录，失败会抛出 AuthenticationException(M3000)
        // 全局异常处理器会自动捕获并返回 M3000
        Long userId = authUtil.requireUserId(token);
        
        List<Order> orders = orderService.getUserOrders(userId);
        return Result.success(orders);
    }
}
```

**优点**：
- ✅ 代码最简洁，无需任何异常处理
- ✅ 全局统一处理，一致性高
- ✅ 语义明确

---

#### 场晩 2：手动处理认证异常（特殊场景）

```java
@RestController
@RequestMapping("/admin/config")
public class ConfigAdminController {
    
    @Autowired
    private AuthUtil authUtil;
    
    @Autowired
    private ConfigService configService;
    
    @PutMapping
    public Result<Void> updateConfig(@RequestBody ConfigDTO config,
                                     @RequestHeader("Authorization") String token) {
        try {
            // 必须是管理员
            authUtil.requireAdmin(token);
        } catch (AuthenticationException e) {
            // 未登录
            return Result.fail(ReasonCode.UNAUTHORIZED);
        } catch (AuthorizationException e) {
            // 非管理员
            return Result.fail(ReasonCode.ADMIN_REQUIRED);
        }
        
        configService.update(config);
        return Result.success(null);
    }
}
```

**适用场景**：
- 需要区分不同的错误类型
- 需要自定义错误响应
- Controller 层局部处理

## 四、最佳实践建议

### 4.1 选择合适的方案

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 个人中心、创建资源 | `authUtil.requireUserId()` | 必须知道用户身份，代码简洁 |
| 浏览公开内容 | `authUtil.getCurrentUserId()` | 未登录也能用，登录后体验更好 |
| 管理员接口 | `authUtil.requireAdmin()` | 区分 M3000 和 M3009，语义明确 |
| 特殊场景需要自定义处理 | try-catch 捕获异常 | 灵活控制错误响应 |

### 4.2 不要混用

```java
// ❌ 不推荐：手动判断 null 后抛异常
Long userId = authUtil.getCurrentUserId(token);
if (userId == null) {
    throw new AuthenticationException(ReasonCode.UNAUTHORIZED);
}

// ✅ 推荐：直接使用 requireUserId()
Long userId = authUtil.requireUserId(token);
```

```java
// ❌ 不推荐：使用 boolean 返回值
if (!authUtil.isCurrentUserAdmin(token)) {
    return Result.fail(ReasonCode.ADMIN_REQUIRED); // 无法区分未登录和非管理员
}

// ✅ 推荐：使用 requireAdmin()
authUtil.requireAdmin(token); // 自动区分 M3000 和 M3009
```

### 4.3 全局异常处理器的注意事项

1. **只处理特定异常**：`AuthExceptionHandler` 只处理 JWT、AuthenticationException、AuthorizationException
2. **不影响业务异常**：其他 RuntimeException 由各自的处理器处理，不会被误捕
3. **添加日志**：所有认证失败都会记录 warn 日志，方便安全审计
4. **Controller 层可选处理**：如果需要自定义错误响应，可以在 Controller 中 try-catch

## 六、常见问题

### Q1: `requireUserId()` 和 `getCurrentUserId()` 有什么区别？

**A**: 
- `requireUserId()`：**必须登录**，token 无效时抛出 `AuthenticationException(M3000)`
- `getCurrentUserId()`：**可选登录**，token 无效时返回 `null`

**选择建议**：
- 如果接口必须知道用户身份（如个人中心），用 `requireUserId()`
- 如果接口未登录也能用（如浏览商品），用 `getCurrentUserId()`

---

### Q2: `requireAdmin()` 如何区分「没登录」和「不是管理员」？

**A**: 
- Token 无效 → 抛出 `AuthenticationException(M3000 UNAUTHORIZED)`
- 已登录但非管理员 → 抛出 `AuthorizationException(M3009 ADMIN_REQUIRED)`

前端可以根据不同的错误码给出不同的提示。

---

### Q3: 为什么不再全局捕获 `RuntimeException`？

**A**: 
- 避免误捕其他模块的业务异常
- 只处理明确的认证相关异常（JWT、AuthenticationException、AuthorizationException）
- 其他 RuntimeException 由各自的处理器处理，保持职责清晰

---

### Q4: 可以在 Service 层使用 AuthUtil 吗？

**A**: 可以，但建议在 Controller 层使用。Service 层应该专注于业务逻辑，认证逻辑应该在 Controller 层处理。

---

### Q5: 为什么要添加 warn 日志？

**A**: 
- **安全审计**：记录谁尝试访问受限资源
- **问题排查**：快速定位认证失败的原因
- **监控告警**：可以基于日志设置告警规则

## 七、总结

### 核心价值

1. ✅ **简化代码**：减少 try-catch 块，代码更简洁
2. ✅ **统一处理**：所有模块使用相同的认证方式
3. ✅ **提高可读性**：方法名清晰表达意图（require vs get）
4. ✅ **易于维护**：认证逻辑集中管理
5. ✅ **灵活选择**：提供三种方法适应不同场景
6. ✅ **安全审计**：添加 warn 日志，方便排查和监控
7. ✅ **精确控制**：区分 M3000（未登录）和 M3009（无权限）

### 推荐使用顺序

1. **必须登录** → 使用 `authUtil.requireUserId()` - 代码最简洁
2. **可选登录** → 使用 `authUtil.getCurrentUserId()` - 灵活可控
3. **管理员校验** → 使用 `authUtil.requireAdmin()` - 语义明确
4. **特殊场景** → try-catch 捕获异常 - 自定义处理

### 关键改进点

| 改进项 | 旧设计 | 新设计 |
|--------|--------|--------|
| 异常处理 | 通过 message 字符串判断 | 使用明确的异常类 |
| 全局捕获 | 捕获所有 RuntimeException | 只捕获特定认证异常 |
| 管理员校验 | 返回 boolean，无法区分错误 | 抛出不同异常，区分 M3000/M3009 |
| 日志记录 | 无 | 所有认证失败都记录 warn 日志 |
| API 设计 | getCurrentUserIdOrThrow() | requireUserId() / requireAdmin() |
